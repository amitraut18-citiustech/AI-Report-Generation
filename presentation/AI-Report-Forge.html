AI Report Forge

Report Migrator Plugin

Phase 1 migrates legacy SSRS reports to web-native HTML with a Claude Code plugin. Phase 2 serves them at runtime with natural-language querying and a local LLM.

Phase 1: Problem → SSRS pain → HTML → Plugin → Migration  ·  Phase 2: Runtime → NL query → Brain → PHI safety
01 · Problem statement

Reporting depends on developers & a report server

Every new or changed report is an IT ticket.

Today

  • SSRS / Crystal reports authored as proprietary RDL by developers.
  • Layout, data source, deploy to a report server for each report.
  • Reports are static snapshots, PDF-centric.
  • Change cycle is slow; business users can't self-serve.

Goal

  • Replace SSRS for on-demand, read-only tabular/summary reports.
  • Reports become static HTML + JS in source control.
  • Reuse the existing .NET data layer — no data rewrite.
  • An AI plugin migrates existing reports automatically.
02 · Why SSRS hurts

Issues with SSRS reports

Runtime & infra

  • Needs an SSRS server or the ReportViewer engine.
  • ReportViewer can't run in-process on .NET 8 (depends on the .NET Framework vbc.exe) — a separate .NET Framework 4.8 side-service is needed just to render RDL.
  • Licensing + server + tooling overhead.

Authoring & output

  • RDL is verbose XML — hard to diff, review, or restyle; needs Report Builder / Visual Studio.
  • Fragile coupling (e.g. a parameter must have a matching layout cell or the RDL is rejected).
  • Output is PDF-like: not responsive, not web-native, limited interactivity.
03 · Solution

HTML-based reports

Self-contained HTML + JS templates that render from an injected data object — the host app owns the data, the template just renders.

Web-native

Responsive, styleable, printable, with interactive filters. No report server needed.

Data contract

The host injects window.REPORT_DATA = { parameters, rows, meta }. The JS never fetches and never touches the DB.

Versioned

Report definitions live as HTML/JS/MD in the repo — reviewable diffs, no server deploy.

Same data, new renderer

The existing .NET data layer produces the rows (already filtered); the host replaces a <!-- REPORT_DATA --> marker in the template with the serialized data and serves a fully self-contained page.

07 · Impact on the app

What changes & what stays the same

✔ Stays as-is

  • The database & schema.
  • The data-access layer (EF Core queries / data services).
  • Business logic & filtering — still applied in .NET.
  • Controllers, routing, authentication.
  • Report parameters and their meaning.

↻ Changes

  • Rendering engine: SSRS RDL + ReportViewer PDF → HTML template populated with REPORT_DATA.
  • A new controller action serves the populated HTML (e.g. HtmlReport).
  • Filters become interactive HTML controls that round-trip via the query string.
  • No SSRS server / ReportViewer dependency on the HTML path.
  • Report definitions live as HTML/JS in git, not .rdl on a server.

Net effect: the data pipeline is untouched; only the presentation layer is swapped — proving SSRS can be replaced without disrupting the app.

Example

"Give me patient data for Austin General" → the brain uses the schema map to know Austin General is a facility, decodes report=patient with filter Facilities.Name contains "Austin General", and the report shows the 3 patients there (not all 8), with an AI narrative.

The LLM replaces the SSRS parameter form — users type questions instead of navigating dropdowns.

10 · Python brain

The brain service (FastAPI)

Endpoints

  • POST /decode-prompt — NL question → report key + table-qualified filters
  • POST /summarize — rows → narrative (+ optional chart)
  • GET /health — readiness: reports, schema tables, PHI markers

Loaded at startup

  • ReportThoughts/*.thought.md — report catalog
  • schema-mapping.json — tables + navigation for cross-table filters
  • phi-markers.json — PHI columns + strategies

Local LLM (primary)

Ollama · qwen2.5:3b — runs on-device, CPU-only; does both decode and summarize.

Cloud fallback

Anthropic Claude API — only if Ollama fails, and only on anonymized data.

11 · PHI safety

PHI never leaves the server in the clear

ROWSReal datanames, DOB, MRN…
→
ANONYMIZEphi-markers.jsonpseudonymize / age-range / redact
→
LLMOllama / Claudesees only pseudonyms
→
RE-MAPLocalpseudonyms → real names

Guarantees

  • Anonymize rows and the question before any LLM call; re-map locally after.
  • The brain refuses to start without its PHI markers.

Guardrails

  • Query spec is signed + encrypted; only allowlisted columns are filterable.
  • Deterministic post-decode fixes; off-topic / destructive questions fail closed; honest errors (no fabricated summary).
12 · Rendering

Two runtime render paths, one data layer

HTML template — SSRS replacement

  • Host injects window.REPORT_DATA: rows + applied filters + AI narrative + optional chart.
  • Web-native, interactive, self-contained.
  • /Reports/HtmlReports

Real SSRS .rdl → PDF

  • Rendered by RdlRenderService (ReportViewer) for legacy parity.
  • Shown in an iframe; Download PDF on every report.
  • /Reports/Index

The same EF Core data + filters feed both paths — proving the HTML report is a drop-in replacement for SSRS.

1) Ollama ollama serve :11434 2) Python brain uvicorn ai_report_forge.api:app --port 8080 :8080 3) RDL service dotnet run (DotNetApp/RdlRenderService) :5250 4) .NET web app dotnet run (DotNetApp/PatientReports) :5282 Open ▶ http://localhost:5282/Reports/HtmlReports

Local & private

LLM runs on-device; SQLite auto-seeds on startup.

No SSRS server

HTML path needs no report server; the RDL path is a self-hosted side-service.

Fully in git

Reports, schema, thought files — all version-controlled.
