1. Introduction & Purpose

This document specifies the requirements for AI Report Forge, a proof-of-concept platform intended to reduce developer dependency for routine reporting by replacing traditional SSRS/Crystal Reports with an AI-assisted, web-native reporting solution. It is the baseline requirements artifact and is written prior to architecture, design, and implementation; all subsequent design and development decisions trace back to the requirements herein.

The requirements use MoSCoW priorities: Must, Should, Could. The keyword “shall” denotes a mandatory requirement.

2. Background & Problem Statement

Today, every new or changed report (SSRS .rdl / Crystal) requires developer effort: authoring proprietary report definitions, designing layouts, configuring data sources, and deploying to a report server. This creates IT backlogs, slows the business, and produces static, PDF-centric output that is not web-native and cannot be self-served. Reports are also difficult to review, diff, restyle, and version-control.

There is a need for a solution that (a) migrates existing reports into maintainable, version-controlled artifacts with minimal manual coding, and (b) lets business users obtain reports on demand — ideally by asking questions in plain language — while reusing the existing data access layer and honoring healthcare data-privacy (PHI/HIPAA) obligations.

3. Goals & Objectives

  • G1. Replace SSRS/Crystal for on-demand, read-only tabular and summary reports.
  • G2. Reduce developer dependency for creating and migrating reports.
  • G3. Let business users retrieve reports via natural-language questions.
  • G4. Reuse the existing .NET data access layer and database — no data-layer rewrite.
  • G5. Keep report definitions as version-controlled, reviewable web artifacts.
  • G6. Guarantee that patient PHI never leaves the server boundary in the clear.

5. Scope

5.1 In scope (PoC)

  • Migration of existing SSRS .rdl (and Crystal doc) reports to static HTML+JS report templates.
  • A runtime web application that serves reports populated with live data.
  • Natural-language question → report selection + filters, via a local LLM or — at the user's explicit choice — a cloud LLM (Claude), with automatic cloud fallback when the local model fails.
  • AI-generated narrative summaries for report results (charts supported behind a configuration flag, disabled by default).
  • PDF export and rendering of the original SSRS definitions for parity.
  • A database schema map and PHI classification driving decoding and anonymization.

5.2 Out of scope (PoC)

  • Scheduled/subscription report delivery (email), drill-down navigation, sub-report authoring UI.
  • Row-level security / per-user data partitioning.
  • Write-back or transactional operations (reporting is read-only).
  • Production-grade multi-tenant scaling and HA.

6. Functional Requirements

6.1 Report migration (build-time tooling)

IDRequirementPriority
FR-1.1The tooling shall analyze an existing report definition (SSRS .rdl, or its host code where the RDL is skeletal) and produce a human-reviewable analysis of its data, business logic, parameters, and layout.Must
FR-1.2A developer shall be able to review and approve the extracted analysis before any report code is generated.Must
FR-1.3The tooling shall generate a static HTML + JavaScript report template (plus documentation) equivalent to the source report.Must
FR-1.4The tooling shall produce a database schema map and a PHI-column classification for use at runtime.Must
FR-1.5The migration tooling shall be host-agnostic (reusable across report projects); application-specific facts shall live in a per-project context artifact.Should
FR-1.6All generated artifacts shall be committed to version control (reviewable diffs, no report-server deploy).Must

6.2 Runtime reporting application

IDRequirementPriority
FR-2.1The application shall present a report catalog and allow a user to generate any available report.Must
FR-2.2The application shall fetch report data using the existing data access layer; the report template shall not access the database directly.Must
FR-2.3The application shall render a report by injecting the fetched data into the generated HTML template.Must
FR-2.4The application shall reuse the existing authentication and authorization model.Must

6.3 Natural-language querying

IDRequirementPriority
FR-3.1The system shall let a user ask a plain-English question and route it to the correct report.Must
FR-3.2The system shall extract structured, table-qualified filters (including cross-table conditions) from the question.Must
FR-3.3The system shall use a database schema map so that terms are resolved to the correct table/column (e.g. a facility name vs. a patient name).Should
FR-3.4When no report matches, the system shall return a clear “unknown / cannot answer” response rather than a fabricated result.Must
FR-3.5Users shall also be able to select a report and set filters manually, without asking a question.Should
FR-3.6Users shall be able to choose which model interprets their question — the local LLM (default) or the cloud LLM — via distinct actions in the UI.Should
FR-3.7The system shall always disclose which model produced an answer (local, cloud, or cloud-as-fallback) alongside the result. (As built: additionally, a Prompt Log page shows, per LLM call, the original question/data side-by-side with the scrubbed/anonymized payload actually sent.)Must

6.4 Filtering

IDRequirementPriority
FR-4.1The system shall apply report filters at the database query level (not by filtering in memory after loading all rows).Must
FR-4.2Only an allowlisted set of fields per report/table shall be filterable; non-reportable and sensitive contact fields shall be blocked.Must
FR-4.3The applied filters shall be displayed to the user (echo / badges) for transparency.Should
FR-4.4Filter definitions should be declarative so that adding/removing a filter does not require rewriting report code.Could

6.5 Output, summaries & export

IDRequirementPriority
FR-5.1The system shall present an AI-generated narrative summary of the report results.Should
FR-5.2The system may present an optional chart when appropriate; any chart specification shall be validated before rendering. (As built: implemented behind a configuration flag, disabled by default to keep LLM responses fast and the PoC focused.)Could
FR-5.3The system shall allow the report to be downloaded as a PDF.Must
FR-5.4The system should be able to render the original SSRS .rdl definition for side-by-side parity with the HTML report.Should

7. Non-Functional Requirements

IDRequirementPriority
NFR-1 · Privacy/PHIDatabase rows shall never leave the server boundary in the clear. Any rows sent to an LLM for summarization (local or cloud) shall first be anonymized (pseudonymization, age-range generalization, redaction, region-only), with known PHI values also scrubbed from the question, and real values re-mapped locally afterwards. Exception: the question-decoding step inherently transmits the question text to the chosen model (filter values must be extracted from it) — never any rows — and any cloud use is disclosed per FR-3.7.Must
NFR-2 · SecurityReporting shall be read-only; queries shall be parameterized; the query specification passed between components shall be tamper-resistant (signed/encrypted); destructive or off-topic requests shall fail closed.Must
NFR-3 · Local-firstThe default language model shall run locally (on-device) so that routine operation requires no external network or per-call cloud cost. A cloud model may be used only (a) on the user's explicit request, or (b) as an automatic fallback when the local model fails — in both cases visibly disclosed in the UI.Must
NFR-4 · ReliabilityIf the language model is unavailable or fails, the system shall degrade gracefully and return an honest error rather than fabricated content; manual report selection shall still work.Must
NFR-5 · PerformancePrompt decoding should complete within ~10s and data retrieval within ~5s on the target hardware; results should be capped to a safe row limit.Should
NFR-6 · MaintainabilityReport definitions shall be plain, version-controlled text (HTML/JS/JSON/Markdown) that can be reviewed via normal code review.Must
NFR-7 · LicensingThird-party libraries shall use permissive open-source licenses (e.g. MIT/Apache); revenue-gated or restrictively-licensed components shall be avoided.Must
NFR-8 · PortabilityNew reports shall be deployable by adding artifacts (definition + template) without rebuilding or redeploying a report server.Should
NFR-9 · UsabilityThe reporting UI shall be web-native, responsive, and printable.Should

8. Data Requirements

The reporting domain covers a healthcare/transplant dataset. The system shall support reporting over related entities including, at minimum: Patients, Facilities, Providers, Transplant Events, Lab Results, Diagnoses, and Medications, with their relationships (e.g. a patient belongs to a facility; transplant events and diagnoses belong to a patient).

  • Each column shall be classified as PHI or non-PHI, with an anonymization strategy for PHI columns.
  • Derived/calculated fields (e.g. age from date of birth, full name, BMI, risk indicators) shall be supported.
  • The database may be seeded with synthetic sample data for the PoC (no production PHI required).

9. Constraints & Assumptions

  • C1. The runtime application is an existing .NET web application; the solution must integrate with it rather than replace it.
  • C2. Rendering the original SSRS .rdl requires the Microsoft ReportViewer engine, which does not run in-process on modern .NET; a separate compatibility service is assumed for that path only.
  • C3. Target hardware is a standard corporate laptop/VDI (CPU-only, ~16 GB RAM, no GPU); LLM choice must fit this budget.
  • C4. Certain network ports may be restricted by corporate policy; the solution must be configurable.
  • A1. Existing parameterized queries and connection security in the .NET app are trusted and reused.

10. Acceptance Criteria (Definition of Done)

  • AC1. An existing SSRS report can be migrated to an HTML report via the tooling, with a reviewed analysis artifact, for at least 3 reports.
  • AC2. A user can submit a plain-English question and receive the correct report, filtered accordingly, with an AI summary.
  • AC3. Filters are demonstrably applied at the database level and change the returned rows.
  • AC4. A report can be exported to PDF, and the original SSRS definition can be rendered for parity.
  • AC5. When the local LLM is stopped, the system either falls back to the cloud model (with the fallback visibly disclosed) or fails honestly — never fabricates data, and never sends database rows externally in the clear.
  • AC6. New reports can be added without modifying/rebuilding a report server.

11. Risks

RiskMitigation
Small local LLM misinterprets questionsSchema-aware prompting; deterministic post-decode corrections; manual filter fallback.
PHI leakage to cloud LLMMandatory anonymization + local re-mapping; refuse to start without PHI markers.
CPU-only inference is slowUse a small quantized model; timeouts; option to select a lighter model.
Legacy RDL rendering incompatible with modern .NETIsolate RDL rendering in a compatibility service; primary path is HTML.

12. Glossary

  • SSRS / RDL — SQL Server Reporting Services and its Report Definition Language (XML).
  • PHI — Protected Health Information (HIPAA).
  • LLM — Large Language Model.
  • Thought file — the reviewable analysis of a report’s logic produced during migration.
  • Query spec — the structured report key + filters decoded from a natural-language question.
